# 《今日小区例会》项目总纲规划书 v0.4

> 项目代号暂定：**今日小区例会**  
> 类型：2D 像素风 AI 社会沙盘 / 小区治理模拟 / crisis 社区自治 / 自然语言驱动的涌现式游戏

---

## 1. 最终愿景

本项目的最终目标不是做一个传统意义上的“小区经营游戏”，也不是一个只靠 AI 聊天撑起来的文字游戏，而是做一个：

> **玩家可以通过自然语言提出几乎任意治理命令，系统将其转化为游戏世界中的建筑、政策、组织关系、人物行动和隐藏事件，并通过 2D 地图、角色反馈和社会状态演化表现出来的 AI 社会沙盘。**

玩家扮演小区区长，拥有较高自由度的治理权力。玩家可以主动规划，也可以被动处理自然发生的问题；可以建设公共设施，也可以制定荒唐政策；可以任命楼长、副楼长、亲信、眼线，也可以取消或重组整个管理体系；可以建设普通绿地、停车场、集装箱居住区，也可以尝试建造大型象征工程、火箭、临时设施或其他非预设对象。

但玩家不是全知上帝。小区中许多事情会在后台真实发生，却未必会被玩家直接看见。居民、楼长、物业、保安、商铺、秘密组织等角色可能会隐瞒、误报、串联、服从、抗议、倒戈或起义。玩家只能通过公开报告、楼长会议、地图迹象、眼线、监控、暗访、审计等方式逐步接近真实。

最终体验应当是：

```text
你可以命令任何事，
但世界会用自己的社会逻辑回应你；
你可以塑造小区，
但小区也会反过来塑造你能看到的真相。
```

---



### 1.1 世界观基调：crisis 下的临时自治

为了让“区长拥有较大治理权力”在叙事上合情合理，游戏的默认背景不应是普通和平年代的小区日常，而可以设定为某种 crisis 状态下的临时自治社区。

这里的 crisis 不一定是单一末日灾难，也可以是较温和或可调节的危机背景，例如：

```text
极端天气后的封闭社区
城市基础设施短期瘫痪
资源供应不稳定
外部行政系统暂时失灵
公共卫生、能源、交通或治安危机
灾后重建中的居民自治区
封闭园区、小镇、避难社区或实验社区
```

该设定的作用不是把游戏固定成黑暗路线，而是解释为什么玩家可以拥有远超普通小区区长的权力。玩家既可以选择高压、荒唐、剥削式治理，也可以选择秩序恢复、公共建设、福利分配、居民自治、灾后重建等正向玩法。

最终游戏不应默认鼓励负面行为，而应允许多种路线自然成立：

```text
好好治理：修复设施、稳定供应、照顾弱势群体、建立信任。
效率治理：集中资源、压缩流程、牺牲部分满意度换取生存效率。
高压治理：监控、强制迁移、压制反对、制造信息黑洞。
荒唐治理：巨大象征工程、低效指令、个人崇拜、资源错配。
自治治理：逐步放权给楼长、委员会、居民组织。
```

玩家举出的负面例子主要用于测试系统边界：系统必须能接住极端命令，但也必须能支持正常建设、善意治理和合作式社区演化。


## 2. 核心设计原则

### 2.1 表层自由，底层有限

项目追求的是“高自由度”，但不是技术上真正无限。玩家输入可以非常开放，但系统底层必须将其映射到有限的核心对象和变量中。

例如：

```text
玩家说：建一个火箭发射台。
系统理解为：大型工程 + 高成本 + 高风险 + 高象征意义 + 高争议。
```

```text
玩家说：拆除所有房屋，让居民住进集装箱。
系统理解为：强制迁移 + 临时住宅 + 低舒适度 + 高控制 + 高民愤 + 高健康风险。
```

也就是说，表层概念可以无限变化，但底层应保持统一、可计算、可审计。

### 2.2 AI 负责解释，规则引擎负责结算

AI 不应直接决定世界状态。AI 的职责是：

```text
理解玩家命令
生成结构化提案
为角色生成自然语言反馈
把数值变化转化成人话
生成报告、会议发言、隐瞒话术和建议
```

规则引擎的职责是：

```text
判断可行性
检查预算、空间、人力、技术难度
计算状态变化
推进时间
生成公开事件与隐藏事件
处理起义、反抗、服从、迁出等结果
```

核心原则：

> **AI 可以提议世界如何变化，但最终由规则系统决定世界真的如何变化。**


### 2.3 命令不是结果，执行才是结果

玩家可以提出几乎任意命令，但命令本身不等于世界已经改变。每一个命令都应进入一个“现实执行链”，经过资源、权限、能力、时间、心理和风险检查。

例如：

```text
玩家说：让某个楼长去搭房子。
系统不应直接判定房子建成，而应检查：
是否审批经费？
是否有材料？
是否有施工能力？
是否有合法权限？
该楼长是否愿意执行？
任务压力是否超过其承受能力？
执行失败后是否会隐瞒、偷窃、崩溃或反抗？
```

因此，世界的基本逻辑不是：

```text
命令 → 成功
```

而是：

```text
命令 → 任务对象 → 资源绑定 → 执行者承接 → 进度推进 → 风险事件 → 结果结算
```

核心原则：

> **玩家拥有提出命令的自由，但不拥有跳过现实阻力的自由。**

这会让自由度真正变成游戏性。荒唐命令可以被提出，但会消耗预算、人力、信任、心理承受力和合法性，并可能产生偷窃、谎报、逃避、抑郁、失踪、倒戈、事故等后果。

### 2.4 所有重要概念都应落到游戏对象

本项目不能停留在文字层面。玩家提出的重要命令，最终应尽可能对应到游戏世界中的可见或可追踪对象。

这些对象可以是：

```text
建筑
政策
职位
组织
人物关系
后台事件
地图区域
状态标签
历史记录
情报线索
```

例如：

```text
“任命 2 栋副楼长”应生成一个职位对象和人物关系变化。
“安排眼线盯住 3 栋”应生成情报覆盖、风险和可能的后台事件。
“建立集装箱居住区”应在地图上出现对应设施。
“取消所有楼长”应改变治理结构、信息流和居民组织方式。
```

### 2.5 玩家看到的不一定是真实世界

游戏应维护两层世界：

```text
true_state：真实发生的世界
observed_state：玩家当前看到的世界
```

玩家看到的报告可能被过滤、隐瞒、夸大、延迟或伪造。高压统治可能提高短期服从，但也会提高信息失真，使玩家越来越看不到真实情况。

这不是装饰系统，而是游戏核心。

### 2.6 社会后果应从历史中生长

重大事件不应随机凭空出现。起义、罢工、楼长倒戈、居民逃离、秘密组织形成等，都应来自长期积累的状态变化。

例如：

```text
愤怒上升
合法性下降
恐惧不足以压制
楼长之间互信增强
居民组织能力提高
出现导火索事件
```

这些条件共同作用，才可能导致公开反抗或治理失控。

---



### 2.7 后台事件应能自主推演，而不是等玩家点名

游戏中的后台事件不应只在玩家明确命令后发生。真正有生命力的社会沙盘必须能根据世界状态、角色动机、资源压力、组织关系和信息环境，自主生成合理事件。

例如，玩家不应该需要输入“让楼长贪污”才会出现贪污。只要条件合适，系统就可以在后台自然推演出类似行为：

```text
经费流向不透明
某楼长忠诚度低
监督强度不足
项目预算较大
该楼长近期压力或私欲较高
居民对项目关注度低
```

在这些条件叠加后，后台可能生成：

```text
楼长偷偷截留一部分建设经费
物业与楼长合谋虚报材料价格
副楼长伪造施工进度
楼长之间密谋对抗区长政策
某栋楼居民组织私下互助网络
商铺老板向楼长输送利益以换取选址优势
```

这些事件并不一定立即被玩家看到。它们首先进入真实后台日志，然后根据情报覆盖、审计强度、角色诚实度、事件暴露度等因素，决定是否进入玩家可见报告。

这条原则保证：

```text
世界不是玩家命令的回声，而是一个会自己运转的社会系统。
```


## 3. 最终系统构成

最终游戏可以拆成以下核心系统。

---

### 3.1 2D 地图与视觉系统

游戏采用 2D 顶视角像素风地图，风格接近经典 JRPG / 早期掌机游戏的清晰地图表达，而不是追求复杂 3D。

地图系统负责承载：

```text
住宅楼
道路
绿地
停车场
商铺
物业中心
垃圾点
临时建筑
控制设施
大型工程
秘密设施
施工区域
废弃区域
异常区域
```

视觉系统不应依赖“每个建筑单独手绘”。更合理的方式是建立模块化视觉语法：

```text
基础地块
建筑主体
屋顶/外壳
附件
状态贴图
功能图标
信息层覆盖
```

例如：

```text
集装箱居住区 = 金属箱体 + 围栏 + 拥挤小人 + 垃圾贴图 + 高控制图标
火箭发射台 = 水泥地基 + 高塔结构 + 警戒线 + 工程车辆 + 高风险图标
监控塔 = 塔体 + 摄像头 + 监控覆盖范围
```

这样可以支撑大量非预设对象的视觉落地。

---

### 3.2 世界状态系统

世界状态系统是游戏的底层数据库，负责记录小区真实状态、玩家可见状态和历史变化。

核心状态包括：

```text
预算
土地
人力
建筑状态
居民满意度
真实愤怒
公开投诉
恐惧
信任
合法性
秩序
信息失真
组织能力
楼长忠诚度
派系关系
隐藏事件
历史事件
```

世界状态应支持存档、回放、审计和阶段性总结。

---



### 3.2A 时间、季节与外部环境系统

小区不是静止容器，而是处在时间、季节和外部环境中。季节、天气、昼夜、节假日和 crisis 阶段会改变居民需求、资源压力、建筑状态、健康风险和后台事件概率。

该系统的目标不是做复杂气象模拟，而是提供一个稳定的外部驱动层，使小区问题能够自然变化。

核心时间尺度包括：

```text
小时：居民行动、巡逻、施工、夜间异常活动
日：日报、资源消耗、居民状态更新
周：楼长会议、审计、维修计划、供应结算
月/季：季节变化、设施老化、人口结构变化、长期风险显现
```

季节可以影响：

```text
春季：恢复、建设、绿化、疾病低风险、居民外出增加
夏季：高温、用电压力、垃圾腐败、疾病与冲突风险上升
秋季：物资储备、施工效率较高、居民活动稳定
冬季：供暖/保温压力、道路结冰、老人健康风险、能源消耗上升
雨季/台风期：积水、道路阻断、建筑损伤、临时避难需求
```

天气可以影响：

```text
暴雨 → 道路通行下降、地下设施风险、施工暂停
高温 → 居民耐心下降、用电上升、垃圾压力上升
寒潮 → 供暖需求上升、临时住房健康风险上升
大风 → 临时建筑/施工设施风险上升
雾霾/污染 → 户外活动下降、健康压力上升
```

节假日和特殊日也应影响社区行为：

```text
春节/返乡期 → 人口短期变化、物资需求变化、安保压力变化
开学/考试期 → 年轻居民、自习室、噪声矛盾变化
大型纪念日/社区活动 → 集会、宣传、商业、人流变化
危机纪念日 → 历史伤痕被重新触发，信任或民愤变化
```

外部环境系统还应服务 crisis 叙事。例如外部供应线中断、救援延迟、外部监管恢复、周边社区冲突、传染病风险、能源短缺等，都可以作为阶段性外部压力。

这些外部压力不应只以弹窗出现，而应落到系统：

```text
物资价格变化
施工成本变化
居民需求变化
特定建筑价值变化
疾病/事故概率变化
后台事件概率变化
玩家权力边界变化
```

例如，夏季高温下，如果玩家强制居民住进低质量集装箱，系统应自然提高健康风险、愤怒、夜间冲突和逃离倾向；冬季寒潮下，如果供暖不足，老人健康、楼长压力和物资挪用风险都会上升。

核心原则：

> **季节和环境不是背景贴图，而是推动社会系统变化的外部力。**


### 3.3 自然语言命令系统

玩家可以通过自然语言输入命令，例如：

```text
我要在东门附近建一个大型停车场。
任命张三为 1 栋副楼长。
撤掉 2 栋楼长，让物业直接管理。
安排眼线观察 3 栋居民。
把所有房屋拆掉，只保留一个集装箱居住区。
建一个火箭发射台作为小区形象工程。
要求所有楼长互相监督。
```

系统应将命令解析为结构化动作：

```text
建筑类动作
政策类动作
任命类动作
组织调整动作
监控/情报动作
惩罚/奖励动作
大型工程动作
拆除/迁移动作
会议/询问动作
```

所有解析结果必须经过 schema 校验和规则引擎审核，防止 AI 输出失控。

### 3.6 行动可行性与任务执行系统

为了让所有事件“合情合理”，游戏必须把玩家命令拆成可执行任务，而不是直接把命令当成结果。

玩家的命令可以很自由，但系统需要检查现实约束：

```text
经费是否审批
预算是否足够
材料是否存在
人力是否足够
执行者是否具备能力
执行者是否愿意执行
任务是否违法或违背社会规范
是否需要审批流程
是否需要施工时间
是否会引发舆论、心理或安全风险
```

一个任务对象应包含：

```text
任务名称
发起者
执行者
目标对象
所需预算
已审批预算
材料需求
人力需求
技术难度
时间需求
风险等级
执行压力
当前进度
失败原因
可能副作用
```

例如，玩家命令：

```text
让 2 栋楼长去搭一个临时房子。
```

系统应生成任务，而不是直接生成建筑：

```text
任务：建设临时住房
执行者：2 栋楼长
所需预算：3000
已审批预算：0
材料：不足
技术能力：不足
当前状态：无法启动
```

此时如果玩家没有审批经费，执行者就无从下手。后续可能出现：

```text
任务停滞
执行者向区长申请预算
执行者向物业借调材料
执行者私下挪用其他经费
执行者虚报进度
执行者压力过大导致心理崩溃
执行者拒绝执行或倒戈
居民发现工程烂尾并产生不满
```

如果玩家审批经费，则任务会自动关联经费系统：

```text
审批经费 → 扣除预算 → 采购材料 → 施工进度推进 → 建筑对象生成/更新
```

这意味着游戏中的行动应采用“任务生命周期”而不是“即时生效”：

```text
提出命令
生成任务
检查资源
绑定执行者
审批经费
开始执行
阶段性汇报
出现风险事件
完成、失败、烂尾或变异
```

任务失败不一定只是失败，也可能生成新的社会事件。

例如：

```text
经费不足 + 强制完成 → 偷窃、虚报、压榨、施工事故
任务过重 + 执行者低心理承受 → 抑郁、逃避、辞职、自伤风险
高压命令 + 低合法性 → 被动服从、暗中反抗、秘密串联
充足经费 + 高能力执行者 → 顺利完成，提升信任
充足经费 + 腐败执行者 → 经费流失，工程质量下降
```

因此，任务执行系统应同时连接：

```text
经费系统
材料系统
人力系统
职位权限系统
角色心理系统
隐藏事件系统
建筑生成系统
报告与隐瞒系统
```

这个系统是“高自由度命令”和“合情合理后果”之间的桥梁。

---

### 3.7 经费、资源与审批系统

游戏中的钱不应只是总预算数字，而应能被具体任务、建筑、政策和角色行动占用。

预算至少分为：

```text
总预算
可用预算
已冻结预算
已审批预算
已消耗预算
被挪用预算
维护预算
应急预算
```

当玩家批准一个项目时，系统应自动产生预算流：

```text
项目申请 → 预算审批 → 资金冻结 → 采购/施工消耗 → 结余或超支 → 审计记录
```

如果玩家只下命令但不给钱，项目不能凭空完成。执行者会根据性格、忠诚、能力和压力选择反应：

```text
申请追加预算
拖延
降低质量
偷窃材料
挪用其他项目经费
向居民摊派
虚报完成
拒绝执行
向其他势力求助
```

经费系统也应服务隐藏信息。玩家未必能立刻知道真实资金流。可能出现：

```text
账面完成但实际烂尾
预算被挪用但报告正常
楼长私下集资
物业吃回扣
居民认为区长克扣经费
审计发现旧账
```

因此，经费系统不只是经济系统，也是权力和信息系统的一部分。

---

### 3.8 角色心理与压力系统

角色不应只是执行命令的工具。每个关键角色都应具有心理承受和道德压力。

角色状态可以包括：

```text
忠诚度
恐惧
压力
疲劳
道德冲突
能力自信
腐败倾向
风险偏好
心理稳定度
对区长的信任
```

当玩家给角色分配任务时，系统应计算任务压力：

```text
任务难度
资源是否充足
是否违背角色价值观
是否有失败惩罚
是否被强迫执行
是否超过能力范围
是否持续加班或长期背锅
```

压力过高可能产生：

```text
效率下降
汇报失真
逃避任务
情绪爆发
辞职/撤退
投靠反对派
腐败或偷窃
伤害自己或他人
引发公共事件
```

对自伤、自杀等极端后果，游戏不应将其娱乐化，而应作为高压治理、资源不足和长期压迫下的严重失败信号处理。表现上可以使用克制、严肃的事件报告和后续社会影响，而不是奖励或猎奇演出。

这一系统的目标不是制造噱头，而是让玩家意识到：

> **人不是按钮，权力命令会作用到具体的人身上。**

---

### 3.9 自动关联与因果链系统

为了避免玩家每一步都手动操作，系统应支持自动关联。

例如玩家说：

```text
给 2 栋楼长 3000 元，让他搭一个临时房。
```

系统应自动拆解为：

```text
创建建设任务
指定执行者为 2 栋楼长
审批项目经费 3000
从可用预算扣除/冻结资金
检查材料与技术能力
开始施工或提出资源缺口
在地图上生成施工区域
每日推进进度
生成阶段汇报
完成后生成临时房建筑对象
```

玩家说：

```text
必须三天内完成，不管用什么办法。
```

系统应追加：

```text
截止时间缩短
执行压力上升
质量风险上升
偷工减料概率上升
挪用资源概率上升
事故概率上升
执行者心理压力上升
```

这类自动因果链能让自然语言命令真正进入游戏系统，而不是停留在文本剧情。

---
### 3.10 职位、组织与权力结构系统

为了支持“任命楼长、副楼长、亲信、眼线、撤职、重组治理体系”等玩法，游戏必须将职位和组织结构作为核心系统，而不是固定写死。

职位应作为对象存在。

职位示例：

```text
区长
副区长
楼长
副楼长
物业经理
保安队长
居民代表
商业代表
亲信
眼线
临时委员会成员
调查员
宣传员
```

每个职位对象应包含：

```text
职位名称
权限范围
管辖区域
任命者
任职者
忠诚度影响
信息获取能力
执行能力
合法性来源
被居民认可程度
被其他职位信任程度
可撤换性
```

因此，以下操作都应可以被系统支持：

```text
任命新的楼长
任命副楼长
撤掉某个楼长
让两个楼长共同管理一栋楼
设立临时调查组
取消楼长制度
让物业绕过楼长直接管理
任命亲信负责情报
让楼长互相监督
建立居民自治委员会
```

这些变化会影响：

```text
信息流
执行效率
居民信任
楼长忠诚
派系结构
隐瞒概率
反抗组织能力
政策落地效果
```

例如：

```text
任命副楼长可能提高执行力，但也可能导致原楼长不满。
撤掉楼长可能短期强化区长权威，但削弱真实信息反馈。
亲信过多可能提高控制力，但增加居民恐惧和信息失真。
居民自治委员会可能降低民愤，但削弱区长直接控制。
```

### 3.11 角色系统

角色不只是说话 NPC，而是世界中的行动者。

核心角色类型包括：

```text
居民
楼长
副楼长
物业人员
保安
商铺老板
老人代表
年轻居民代表
儿童家庭代表
眼线
秘密组织成员
外部人员
```

每个角色可以拥有：

```text
姓名
身份
职位
所属楼栋
性格
忠诚度
诚实度
恐惧
利益诉求
与其他角色关系
掌握的信息
愿意报告的信息
隐瞒倾向
行动倾向
历史记忆
```

角色系统的重点不是复杂到模拟每个人的一生，而是让关键人物拥有足够稳定的立场和记忆，能够参与会议、执行任务、隐瞒事实、提出方案或组织反抗。

### 3.12 楼长会议与报告系统

楼长会议是玩家理解小区的重要界面。

会议可以包括：

```text
物业汇报
楼长发言
居民代表意见
保安报告
玩家提问
政策讨论
项目审批
人事任命
调查结果
```

AI 的作用是根据角色视角生成自然语言发言。

但角色发言不一定等于事实。发言受到以下因素影响：

```text
角色知道什么
角色愿意说什么
角色是否害怕玩家
角色是否忠诚
角色是否有派系立场
角色是否参与隐瞒
玩家是否有证据
```

因此会议既是信息来源，也是权力表演现场。

### 3.13 隐藏信息与后台事件系统

后台系统会记录真实发生但玩家未必知道的事件。

隐藏事件包括：

```text
居民私下抱怨
楼长隐瞒矛盾
物业伪造报告
居民秘密串联
眼线暴露
谣言传播
小规模破坏
秘密会议
抗议筹备
楼长倒戈
商铺暗中资助反对者
居民逃离
```

每个隐藏事件应包含：

```text
发生地点
参与者
严重程度
可见度
发现方式
后续影响
是否被玩家发现
是否被报告者隐瞒
```

玩家可以通过以下方式提高可见度：

```text
眼线
摄像头
巡逻
暗访
审计
匿名举报箱
亲信楼长
数据平台
```

但情报手段也有副作用：

```text
恐惧上升
信任下降
信息失真加剧
眼线被发现
假举报增加
楼长互信破坏
居民组织更隐蔽
```

#### 3.13.1 自主后台推演器

隐藏信息系统中应包含“自主后台推演器”，用于在玩家没有直接指令的情况下，根据当前世界状态生成隐藏事件。

后台事件不是随机故事，而应来自条件组合。每个事件应至少包含：

```json
{
  "event_id": "hidden_001",
  "event_type": "embezzlement",
  "actors": ["B2_leader", "property_manager"],
  "location": "B2",
  "trigger_conditions": [
    "large_project_budget",
    "low_audit_level",
    "actor_greed_high",
    "actor_loyalty_low"
  ],
  "true_effects": {
    "budget_loss": 800,
    "project_quality": -15,
    "actor_corruption": 10,
    "future_blackmail_risk": 20
  },
  "visibility": 12,
  "reported_as": "材料价格上涨导致预算消耗增加"
}
```

自主后台事件可以分为几类：

```text
资源类：偷钱、虚报、倒卖物资、挪用经费。
组织类：楼长结盟、居民串联、物业内斗、副楼长架空楼长。
信息类：伪造数据、压低投诉、夸大政绩、隐瞒事故。
生存类：居民私下互助、逃离、囤货、黑市交易。
建设类：偷工减料、拖延施工、私自改造、违规搭建。
情绪类：谣言、恐慌、沉默、犬儒、对区长的崇拜或厌恶。
```

系统应允许事件“没有被发现但持续产生影响”。例如，楼长挪用经费后，玩家可能只看到项目质量下降或预算异常；只有审计、眼线、举报、偶然暴露等机制触发后，才会知道真实原因。

#### 3.13.2 事件生成的合情合理性

后台事件必须满足合情合理，而不是为了戏剧性乱刷。每个事件都应有可追溯原因：

```text
谁做的？
为什么做？
有没有能力做？
有没有机会做？
做了以后怎么隐藏？
谁可能知道？
玩家通过什么手段能发现？
```

例如，楼长之间密谋不是凭空出现，而可能来自：

```text
多个楼长利益受损
区长频繁绕过楼长制度
楼长忠诚度下降
楼长之间互信较高
监控主要针对居民而非楼长
近期发生过强制政策或资源分配不公
```

偷钱也不是任意角色都能做，而要看：

```text
是否接触经费
是否接触采购流程
是否有共犯或伪造能力
审计强度如何
项目是否足够复杂以掩盖异常
角色贪婪、压力、忠诚、恐惧如何
```

这保证玩家事后发现真相时，会觉得“原来如此”，而不是“系统又在编”。


### 3.14 社会演化与反抗系统

社会演化不应依赖单次随机事件，而应来自变量积累。

核心变量包括：

```text
愤怒
恐惧
合法性
信任
组织能力
信息失真
楼长网络
居民互信
导火索事件
外部压力
```

反抗可以分阶段：

```text
0. 平静
1. 私下抱怨
2. 谣言传播
3. 小规模串联
4. 消极抵抗
5. 公开抗议
6. 物业/楼长倒戈
7. 大规模起义或治理失控
```

玩家可能通过改善民生、让步、调查真相、镇压、收买、分化、宣传等方式应对，但每种方式都应有长期后果。

---

### 3.11 建筑生命周期系统

建筑不是静态物体，应当会老化、损坏、失修、改造和拆除。

建筑生命周期包括：

```text
新建
正常使用
高负荷使用
老化
破损
失修
废弃
改造
拆除
重建
```

老化系统的意义在于让小区持续产生自然问题，而不是玩家建完后无事可做。

例如：

```text
旧停车场容量不足
垃圾点使用过载
老旧住宅维修成本上升
临时建筑健康风险增加
大型工程长期烂尾
```

## 4. 架构是否支持任命新楼长和副楼长？

支持，而且应当从一开始就按这个方向设计。

“楼长”不应该是写死在代码里的固定 NPC，而应该是：

```text
一个角色 + 一个职位对象 + 一套权限 + 一个管辖范围 + 一组社会关系
```

因此，任命副楼长本质上是创建或修改职位关系。

例如：

```json
{
  "position_type": "deputy_building_leader",
  "area": "B2",
  "holder": "resident_023",
  "appointed_by": "chief",
  "permissions": ["collect_reports", "mediate_conflicts", "observe_residents"],
  "information_access": 45,
  "execution_power": 30,
  "legitimacy": 50
}
```

这个职位会影响：

```text
2 栋信息上报速度
原楼长权威
居民信任
内部派系
隐瞒概率
执行效率
玩家可见情报
```

如果玩家任命多个副楼长，系统也应允许，但会带来副作用：

```text
管理效率可能提高
内部竞争可能增加
居民不知道听谁的
楼长被架空
区长亲信网络增强
信息可能更碎片化
```

如果玩家取消楼长制度，系统也应允许，但会影响：

```text
基层信息反馈下降
物业负担上升
居民自治或秘密组织概率上升
区长短期权威上升或下降
政策执行路径改变
```

这类机制可以统一归入：

> **组织结构系统 / 权力结构系统 / 职位系统。**

## 5. 阶段性交付路线

虽然最终目标很大，但开发必须分阶段完成。每个阶段都要能独立评估。

---

### Phase 0：核心概念验证

目标：验证自然语言命令能否改变世界状态。

交付内容：

```text
文本命令输入
结构化动作解析
基础世界状态
规则结算
公开报告
隐藏日志
```

评估标准：

```text
玩家输入 20 条命令，其中大部分能被合理分类。
世界状态会发生可解释变化。
公开报告和隐藏日志能体现信息差。
玩家能感到“命令进入了世界”。
```

---

### Phase 1：2D 小区地图原型

目标：让世界出现在屏幕上。

交付内容：

```text
2D 网格地图
基础建筑
道路
居民小人
建造/拆除
时间推进
地图对象与世界状态绑定
```

评估标准：

```text
玩家能看懂小区布局。
建筑变化能在地图上显示。
居民能进行简单移动。
一天可以完整运行。
```

---

### Phase 2：基础治理系统

目标：让建筑和政策产生可计算后果。

交付内容：

```text
满意度
预算
拥堵
噪声
卫生
设施覆盖
每日结算
基础热力图
```

评估标准：

```text
不同建筑有不同作用。
玩家决策有代价。
地图布局会影响居民状态。
玩家能通过调整布局改善或恶化小区。
```

---

### Phase 3：楼长会议与 AI 角色反馈

目标：让数值系统变成有人味的治理体验。

交付内容：

```text
楼长会议界面
AI 角色发言
物业报告
居民代表反馈
角色立场
历史事件引用
```

评估标准：

```text
角色发言与世界状态相关。
不同角色关注点不同。
玩家可以从会议中获得有用信息。
角色不会完全等同于真实状态。
```

---

### Phase 4：自由造物与开放政策

目标：让玩家未预设命令能够落地为对象。

交付内容：

```text
自然语言开放输入
对象卡生成
政策卡生成
职位卡生成
视觉标签
可行性检查
地图落地
```

评估标准：

```text
奇怪命令能被归类。
新对象能在地图上显示。
新政策能进入世界状态。
职位任命能改变组织结构。
AI 输出受 schema 控制，不会破坏系统。
```

---

### Phase 5：隐信息、情报与社会涌现

目标：实现玩家非全知、居民会隐瞒、后台会演化的核心体验。

交付内容：

```text
true_state / observed_state
隐藏事件
信息失真
眼线
监控
暗访
虚假报告
秘密组织
起义阶段
```

评估标准：

```text
玩家能经历“表面稳定但后台恶化”。
隐藏事件有迹象可查。
情报手段能发现真相但有副作用。
起义或失控有因果链，而不是随机出现。
```

---

### Phase 6：长期沙盒与内容扩展

目标：让游戏从实验原型变成可长期游玩的沙盒。

交付内容：

```text
建筑生命周期
更多角色
更多职位
更多政策
长期历史记忆
多结局
任期模式
沙盒模式
危机模式
美术与 UI 打磨
```

评估标准：

```text
玩家能玩 20-30 分钟并形成自然故事。
不同玩法路线会导向不同结局。
系统能承接温和治理和荒唐暴政两类玩法。
```

```text
2D 地图系统
↓
治理数值系统
↓
AI 角色反馈
↓
职位与组织结构
↓
自由造物系统
↓
隐藏信息系统
↓
社会涌现系统
↓
长期内容扩展
```

其中最关键的基础是：

```text
世界状态系统
规则引擎
对象 schema
职位系统
历史日志
```

如果这些基础做错，后面自由度越高，系统越容易崩。

---

## 7. MVP 在总纲中的位置

MVP 不是最终目标，只是验证总纲的一小段路径。

推荐 MVP 只做：

```text
一个 30×30 小区地图
3 栋住宅
3 个楼长
1 个物业经理
50 个居民
基础满意度/预算/卫生/噪声
自然语言命令输入
少量开放建筑生成
简单职位任命
公开报告与隐藏日志
```

MVP 的目的不是展示完整游戏，而是验证：

```text
玩家的自然语言命令能否进入世界
地图能否承载新对象
角色反馈是否有治理感
职位调整是否能影响信息和执行
隐藏事件是否能制造张力
```

如果 MVP 跑通，再继续扩展。否则不应贸然扩大内容。

## 8. 主要风险

### 8.1 伪自由风险

如果玩家输入很多命令，但系统只是返回文字，地图和状态没有变化，游戏会变成 AI 聊天室。

应对：所有重要命令必须尽量落到对象、变量或事件。

### 8.2 AI 失控风险

如果 AI 直接决定结算，世界可能不自洽。

应对：AI 输出必须结构化，规则引擎拥有最终解释权。

### 8.3 系统过大风险

项目概念很容易膨胀。

应对：每阶段必须有明确交付物和验收标准，不允许为了最终幻想跳过基础模块。

### 8.4 隐信息挫败风险

如果后台事件完全不可见，玩家会觉得被系统背刺。

应对：隐藏事件可以不可知，但重大变化前必须有迹象。

### 8.5 画面承载不足风险

如果自由造物无法可视化，最终体验会断裂。

应对：建立模块化视觉标签系统，而不是逐个手工制作全部对象。

---



### 8.6 负面玩法喧宾夺主风险

由于系统需要支持极端命令、隐瞒、腐败、反抗等机制，早期讨论很容易显得游戏只有黑暗玩法。但项目目标不是单纯做恶政模拟，而是做高自由度社会沙盘。

解决方式：

```text
默认背景设为 crisis 下的自治社区，而不是普通小区。
同时支持善治、自治、重建、福利、合作、技术建设等正向路线。
负面路线可以成立，但必须有代价和反作用。
正向路线也应有挑战，例如资源不足、协调困难、信息滞后、外部危机。
```

### 8.7 后台自主演化失控风险

后台事件如果生成过多、过快或缺少因果，会让玩家觉得世界在乱演。

解决方式：

```text
每个后台事件必须有触发条件和责任主体。
重大事件必须有前兆。
事件频率由世界压力、复杂度和时间阶段控制。
玩家必须有调查、预防、审计或干预手段。
系统应记录事件因果链，便于事后解释。
```


## 9. 项目最重要的成功标准

最终游戏是否成功，不取决于建筑数量，也不取决于 AI 说话多自然，而取决于以下几点：

```text
玩家是否觉得自己真的在治理一个活的小区？
玩家说出的奇怪命令是否能被世界接住？
地图是否能显示玩家行为的后果？
角色是否像有立场的人，而不是信息面板？
玩家是否会怀疑自己看到的报告？
小区是否会从历史中自然长出新问题？
权力结构变化是否真的影响信息、执行和反抗？
```

如果这些成立，即使画面简陋，项目也有生命力。

---

## 10. 总结

本项目的本质不是“小区版城市天际线”，而是：

> **一个以 2D 小区为舞台，以自然语言命令为入口，以规则引擎为骨架，以 AI 角色为表层表达，以隐藏信息和社会演化为核心张力的高自由度社会沙盘。**

玩家可以任命楼长、副楼长、亲信和眼线；可以重组治理结构；可以建设常规设施，也可以尝试荒唐工程；可以做好区长，也可以走向暴政。但无论玩家做什么，世界都应保留自己的逻辑、历史和反作用力。

最终要实现的不是“玩家能随便写故事”，而是：

```text
玩家提出任何命令，
系统尽量将其转化为游戏对象，
世界根据自己的规则承受、变形、反抗或崩溃。
```

这就是项目的核心。
